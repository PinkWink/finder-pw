PROJECT LOG · v0.3.1

PWFinder — 4분할 파일 탐색기 작업 히스토리

우분투 기본 탐색기를 대체할, 화면을 최대 4분할로 나누고 각 패널에 색상·이름·git 상태·터미널·트리 사이드바를 둔 데스크톱 탐색기. Tauri 2 + React 18 기반으로 처음부터 빌드, GitHub 게시, APT 자동 배포까지 완료.
날짜 2026-05-07 ~ 2026-05-08
저장소 github.com/PinkWink/finder-pw
APT pinkwink.github.io/finder-pw
최신 v0.3.1 트리 UX patch

개요

사용자(PinkWink)는 우분투 기본 파일 매니저(Nautilus)의 화면 분할 미지원이라는 한계를 해결하기 위해, 최대 4개의 분할창을 가지고 각 분할창에 색상·이름을 부여하고 git 상태·터미널 단축까지 통합된 탐색기 도구를 만들고자 했다. 첫 세션(2026-05-07)에 Rust·Tauri·React 셋업 → 기능 구현 → GitHub 게시 → 자체 APT 배포 → 4번의 릴리스(v0.1.0 → v0.2.0)까지 진행됐고, 이어서 v0.2.1(다중 선택)과 v0.3.0(ARM64 Linux 멀티아키 + 분할선 드래그 리사이즈 + 컬럼 정렬 + 파일명 말줄임)이 추가됐다.

  • (1) 좌측 트리는 기본적으로 사용자의 홈(/home/pw) 하위만 보여서 /bin·/boot·/etc 등 시스템 디렉토리로 어수선해지지 않아야 함. "특별히 클릭하지 않는 한"은 = 명시적으로 home 밖 경로로 나가지 않는 한.
  • (2) pane에서 "터미널 열기"를 선택했을 때 해당 pane의 경로에서 터미널이 열려야 함. 헤더 ▶_ 버튼은 이미 그렇게 동작 중이지만 "선택"이라는 단어는 우클릭 메뉴 의미를 함의 → 컨텍스트 메뉴에도 추가하기로 결정.
  • (3) 활성 pane이 가리키는 폴더는 트리에서 항상 펼쳐져 보여야 함. 기존엔 같은 path 재방문 시 expand가 한 번만 일어나서, 사용자가 접었다가 다시 돌아오면 자동으로 다시 펼쳐지지 않았음 — "항상"의 강조는 그 동작을 바꾸라는 뜻.

대응 (1) 트리 루트 동적화 — App.tsx · TreeSidebar.tsx

  • App.tsx에 home 상태 추가, get_home_dir 결과를 보관 → TreeSidebar에 homeDir prop 전달.
  • TreeSidebar 내부에서 활성 path가 home 아래면 트리 루트를 homeDir로, 아니면 "/"로 useMemo 계산. isUnder() · ancestorsOf(path, root) 헬퍼로 루트 기반 조상 계산을 일반화.
  • 결과: 첫 화면에서 /home/pw가 트리 루트, 시스템 디렉토리는 가려짐. pane이 /etc 등 home 밖으로 가면 트리 루트가 자동으로 "/"로 폴백되어 해당 폴더가 노출됨.

대응 (2) 활성 폴더 항상 펼침

기존 lastAutoExpand ref 제거. activePath / root 변경 시 매번 조상들을 expanded Set에 add. 사용자가 명시적으로 토글했더라도 다음 pane 이동 시 다시 펼쳐짐 — "항상"에 부합. setExpanded가 같은 내용이어도 새 Set 참조이지만 useEffect 의존성에 expanded가 빠져있어 무한 루프는 발생하지 않음.

대응 (3) 컨텍스트 메뉴에 터미널 열기 추가

Pane.tsx의 buildContextMenuItems()에 "Open terminal here" 항목을 separator와 함께 Move-to-trash 직전에 삽입. 콜백은 기존 openTerminal()(invoke("open_terminal", { path: config.path })) 재사용. 헤더 ▶_ 버튼은 그대로 유지 — 두 진입점 모두 같은 동작.

지금 나는 이 스크린샷에서 오른쪽 make_confluence_page폴더를 선택했는데, 파일 네비게이션은 그게 안보여, 내가 선택한 폴더가 네비게이션에서도 선택은 되었지만 스크롤이 되지 않은거지

지시 이해

오른쪽 pane이 /home/pw/isaac/make_confluence_page로 갔고 트리에서 ancestor 펼침과 active 클래스 부여까지는 됐지만, 트리 사이드바 컨테이너의 scrollTop이 그대로라서 활성 노드가 뷰포트 아래로 밀려 보이지 않음. "선택은 되었지만 스크롤이 되지 않은거지"는 정확한 자기 진단 — 자동 스크롤만 추가하면 되는 상황.

대응 — 자동 스크롤 추가

  • 각 tree-node div에 data-tree-path={path} 속성 부여.
  • TreeSidebar에 bodyRef(트리 본문 컨테이너) 추가.
  • 새 useEffect([activePath, expanded, childrenMap]): CSS.escape로 안전한 셀렉터를 만들어 querySelector로 활성 엘리먼트 탐색. offsetTop · offsetHeight ↔ scrollTop · clientHeight 비교 → 화면 밖일 때만 scrollIntoView({ block: "nearest" }).
  • 이미 가시 영역 안이면 스크롤 무동작 → 활성 pane을 단순 클릭만 해도 시야가 점프하는 부작용 없음.
  • 의존성에 childrenMap 포함 — 비동기 폴더 로딩이 끝나 활성 노드가 비로소 DOM에 그려진 직후의 시점도 잡힘.
